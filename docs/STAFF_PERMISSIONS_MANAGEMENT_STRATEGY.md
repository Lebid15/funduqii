# Staff + Permissions Management — Phase 11 Strategy

هذه الوثيقة تشرح قرارات Phase 11: واجهة إدارة موظفي الفندق وصلاحياتهم المرنة،
مبنية بالكامل فوق أساس Phase 2 (User / Hotel / HotelMembership /
HotelPermissionGrant / Permission registry) — **بدون أي نظام RBAC جديد وبدون
أي نماذج جديدة**.

## لماذا لا نعتمد أدوارًا ثابتة؟

تسميات مثل Receptionist / Housekeeping / Accountant تختلف من فندق لآخر، وتتغير
مهامها مع الوقت، ويحمل الشخص الواحد أكثر من قبعة في الفنادق الصغيرة. ربط
الوصول بدور ثابت يعني إما أدوارًا كثيرة جامدة أو استثناءات لا تنتهي. لذلك:

- **مصدر الحقيقة الوحيد للوصول**: عضوية نشطة في الفندق + منح صلاحيات صريحة
  (`section.operation`) لكل عضوية.
- **المدير** (membership_type=manager) يملك كل صلاحيات الفندق بحكم النوع —
  بلا منح.
- **job_title وصفي فقط**: تسمية حرّة تُعرض في القوائم ولا تُقرأ في أي فحص
  وصول إطلاقًا (اختبار صريح يثبت أن "Accountant / Finance Manager" لا يمنح
  `finance.view`).
- **قوالب البدء (Presets) أُجّلت عمدًا**: كانت ستضيف طبقة UI/ترجمة تعقّد
  المصفوفة دون قيمة جوهرية — المدير يحدد المنح مباشرة من المصفوفة، وأي قالب
  مستقبلي سيكون مجرد ملء أولي لمربعات الاختيار ثم منحًا عادية.

## البنية

- **لا نماذج جديدة**: `apps/staff` تطبيق API رقيق فوق tenancy + rbac.
- أُضيفت حقول **وصفية** فقط إلى `HotelMembership` (هجرة tenancy.0002):
  `job_title` · `staff_code` · `notes` · `deactivated_at` ·
  `deactivation_reason` · `created_by` · `updated_by`.
  (`joined_at` = `created_at` الموجود؛ `phone` موجود على User؛ `is_active`
  موجود.)
- سجل الصلاحيات المركزي اكتمل بقسم `staff`:
  view / create / update / deactivate / permissions_view / permissions_update
  (+`manage` القديمة موثّقة كـ vestigial).
- كل الكتابة عبر `apps/staff/services.py` حصرًا، وإدارة المنح تستدعي خدمات
  rbac القائمة (`grant/revoke/get_hotel_permissions`).

## دورة حياة الموظف

- **إنشاء مستخدم جديد** (الخيار A): بريد فريد عالميًا (`email_already_registered`
  409 مع اقتراح الربط بدلًا منه)، كلمة مرور مؤقتة تمر بـ
  `validate_password`، تُخزَّن مجزأة ولا تعود في أي استجابة أبدًا.
  **لا يوجد إرسال بريد** (مؤجل عمدًا): كلمة المرور تُسلَّم للموظف خارج النظام —
  موثّق في الواجهة نفسها.
- **ربط مستخدم موجود** (الخيار B): بالبريد؛ عضوية مكررة →
  `membership_already_exists` 409. **مالك المنصة لا يُربط أبدًا**
  (`platform_owner_not_manageable`) — قرار موثّق: نطاق المنصة ونطاق الفندق
  منفصلان تمامًا، ولا يصبح المالك موظف فندق خلسة.
- **تعديل**: حقول وصفية فقط (الاسم/الهاتف على User + المسمى/الرمز/الملاحظات
  على العضوية). البريد هوية دخول ثابتة؛ نوع العضوية والتفعيل والمنح لها
  مسارات خاصة محروسة (ترقية/تنزيل manager↔staff خارج نطاق المرحلة — مؤجلة).
- **تعطيل/إعادة تفعيل** بدل الحذف: لا DELETE إطلاقًا؛ المعطّل يفقد الوصول
  فورًا (`membership_inactive` عند حل السياق — سلوك Phase 2 القائم) ويحتفظ
  بسجله ومنحه كاملة حتى إعادة التفعيل.
- **حماية آخر مدير**: تعطيل آخر مدير نشط → `last_manager_protected` 409 —
  الفندق لا يُترك بلا مسؤول قادر على الإدارة أبدًا. (سحب صلاحيات الإدارة من
  مدير مستحيل بنيويًا: المدير لا يعتمد على المنح أصلًا.)
- **إعادة تعيين كلمة مرور محلية** (اختياري المواصفة — نُفِّذ): المدير أو حامل
  `staff.update` يضع كلمة مؤقتة جديدة تُسلَّم خارج النظام؛ لا بريد.

## تعديل المنح (المصفوفة)

- `PUT /staff/{id}/permissions/` استبدال كامل transaction-safe: حذف الزائد +
  `get_or_create` للجديد (لا تكرار)؛ أي كود غير موجود في السجل →
  `unknown_permission` ولا يتغير شيء.
- **حارس التصعيد**: غير المدير لا يمنح — لنفسه أو لغيره — صلاحية لا يملكها
  هو (`permission_escalation_blocked` 403). الإزالة حرة دائمًا.
- **منح المدير غير قابلة للتعديل** (`manager_permissions_not_editable`) — يملك
  كل شيء بحكم النوع، وأي منح عليه بيانات ميتة توحي زورًا بأنها تتحكم.
- الواجهة تحذّر عند تعديل المستخدم الحالي لنفسه، والباكند لا يعتمد على
  الواجهة في شيء.

## السايدبار والمسارات المحمية

- endpoint جديد خفيف `GET /staff/my-permissions/` (عضوية فقط — عن الذات):
  `{is_manager, permissions[]}` عبر `get_hotel_permissions` القائمة.
- `HotelAccessProvider` يحمّلها مرة لكل جلسة قشرة، وخريطة مركزية واحدة
  (`hotelRouteAccess.ts`) تربط كل مسار بأكواد any-of:
  front-desk→stays.view · reservations→reservations.view · guests→guests.view ·
  finance→finance.view|expenses.view · services→services.view|service_orders.view ·
  operations→housekeeping.view|maintenance.view|lost_found.view ·
  staff→staff.view · rooms→rooms.view · settings→settings.view.
- **السايدبار** يعرض فقط روابط الأقسام المسموحة (المدير يرى الكل؛ أثناء
  التحميل لا يُعرض شيء بدل وميض روابط ممنوعة).
- **حارس المسار** (`HotelRouteGuard`) يعرض «الوصول مرفوض» واضحة عند الدخول
  اليدوي لمسار بلا صلاحية.
- **الإخفاء تجميلي بطبيعته**: كل API يفرض الصلاحيات بنفسه بمعزل عن الواجهة
  (اختبارات صريحة: بلا `finance.view` تُرفض المالية 403 مهما أُخفي الرابط).

## العزل والتعليق

- كل الاستعلامات مقيدة بـ `hotel` من السياق؛ عضويات فندق آخر 404/غير مرئية؛
  المُسند/المستهدف يجب أن يكون من نفس الفندق.
- الفندق المعلّق: القراءة (قائمة/تفاصيل/سجل/نظرة عامة/my-permissions) متاحة؛
  وكل كتابة — إنشاء/ربط/تعديل/تعطيل/تفعيل/منح/إعادة كلمة مرور — ترفض بـ
  `hotel_suspended`.
