# Housekeeping + Maintenance + Lost & Found — Phase 10 Strategy

هذه الوثيقة تشرح قرارات Phase 10: العمليات التشغيلية اليومية المرتبطة بالغرف —
مهام التنظيف، طلبات الصيانة، والمفقودات — وكيف تتكامل مع حالة الغرفة (Phase 5)
وتدفق المغادرة (Phase 7) دون كسر أي قاعدة قائمة.

## لماذا Phase 10 ليست Shifts / Daily Close؟

- **Shifts وStaff scheduling** تحتاج نموذج دوام وموارد بشرية كاملًا (حضور،
  مناوبات، بدلات) — نطاق مختلف تمامًا عن «مهمة تنظيف لغرفة».
- **Daily Close** عملية مالية (إقفال يومية الصندوق) تخص Finance وليست عمليات
  الغرف؛ بناؤها هنا كان سيخلط المال بالتشغيل.
- Phase 10 تبني الدورة الأصغر المفيدة فورًا: غرفة تتسخ → تُنظَّف → تُتاح؛
  عطل يُبلَّغ → يُصلَّح → تُحرَّر الغرفة بقرار صريح؛ غرض يُفقد → يُخزَّن →
  يُسلَّم أو يُتلَف. كل ما عداها مؤجل عمدًا.

## البنية: `apps/operations`

تطبيق مستقل واحد يضم المجالات الثلاثة (بدل ثلاثة تطبيقات صغيرة متشابكة):
- الموديلات لا تعيش داخل `rooms` أو `stays` — الغرف ملكية Phase 5 والإقامات
  ملكية Phase 7؛ العمليات *تشير* إليها فقط.
- لا علاقة لأي شيء هنا بـ Finance — لا توجد كتابة مالية إطلاقًا.
- كل الكتابة تمر عبر `apps/operations/services.py` (المسار الوحيد)، والـ views
  لا تلمس الحالة مباشرة.

## النماذج

### HousekeepingTask
مهمة تنظيف/تجهيز لغرفة: `task_number` (HK00001، فريد داخل الفندق)، الغرفة
(مطلوبة عند الإنشاء؛ FK بـ SET_NULL كي يبقى التاريخ لو حُذفت الغرفة لاحقًا)،
إقامة اختيارية، نوع (checkout/daily/deep/inspection/other)، أولوية، مُسند إليه،
طوابع زمنية لكل انتقال، وسبب إلزامي للإلغاء.

**سير الحالة**: `pending → assigned → in_progress → completed`، والإلغاء من أي
حالة نشطة. نقطة `status/` العامة تسمح فقط بالتقدم الأمامي غير النهائي؛
`complete/` و`cancel/` نقطتان مخصصتان لأن لكل منهما مدخلات إلزامية
(علم إتاحة الغرفة / سبب الإلغاء).

### MaintenanceRequest
طلب صيانة: `request_number` (MT00001)، غرفة/إقامة اختياريتان، عنوان ووصف،
تصنيف وأولوية، `affects_room_availability` + `room_block_status`
(maintenance / out_of_service)، مُسند إليه، ملاحظات حل، وطوابع زمنية كاملة.

**سير الحالة**: `open → assigned → in_progress → resolved → closed`، والإلغاء
(بسبب) من الحالات المفتوحة فقط. `resolve/` و`close/` و`cancel/` نقاط مخصصة.

### LostFoundItem
غرض معثور عليه: `item_number` (LF00001)، عنوان/وصف/تصنيف، مكان العثور
والتخزين، روابط اختيارية بغرفة/إقامة/نزيل، بيانات المستلم، وطوابع زمنية.

**سير الحالة**: `found → stored → claimed → returned → closed` أو
`found/stored → disposed → closed`. نقطة `status/` العامة تسمح فقط بـ
found→stored؛ والباقي عبر `claim/` و`return/` و`dispose/` و`close/`.

**قواعد صارمة**: التسليم (`returned`) يتطلب اسم مستلم أو نزيلًا مرتبطًا
(`claimant_required` 422)؛ الإتلاف يتطلب سببًا (`disposal_reason_required`)؛
لا صور ولا ملفات ولا باركود في هذه المرحلة (مؤجلة عمدًا).

### سجلات الحالة (Status Logs)
ثلاثة نماذج خفيفة (`HousekeepingTaskStatusLog` / `MaintenanceRequestStatusLog`
/ `LostFoundItemStatusLog`) تسجل: من، متى، من أي حالة إلى أي حالة، وملاحظة.
هي **سجلات تشغيلية فقط** وليست Audit Log عامًا (ذلك مؤجل ضمن حوكمة 1.7)،
وتُعرض آخر 10 إدخالات في تفاصيل كل سجل.

### الترقيم (HK/MT/LF)
`OperationsNumberSequence` عدّاد لكل (فندق، نوع) مع `select_for_update` داخل
معاملة — نفس نمط Phase 8/9 المجرَّب، ومنفصل عن التسلسلين المالي والخدمي كي
لا تختلط الأرقام. التوليد على الخادم حصرًا؛ الواجهة لا ترسل أرقامًا أبدًا.

## تكامل حالة الغرفة (النقطة الحساسة)

**القاعدة الذهبية**: لا يكتب `apps/operations` حالة الغرفة مباشرة أبدًا؛ كل
تغيير يمر عبر `apps.rooms.services.change_room_status` — المسار المضبوط من
Phase 5 الذي يتحقق ويسجّل في `RoomStatusLog`.

- **لا `occupied` إطلاقًا**: الإشغال يبقى مشتقًا من إقامة `in_house`
  (قاعدة Phase 7)، ولوحة الغرف تعرضه كشارة «إقامة حالية» منفصلة عن حالة الغرفة.
- **التنظيف**:
  - بدء المهمة يجعل الغرفة `cleaning` فقط إن كانت `dirty`/`available`
    (لا يلمس غرفة محجوبة).
  - الإكمال مع «إتاحة الغرفة» يُرفض بـ `room_blocked_by_maintenance` (409) إذا
    كانت الغرفة `maintenance`/`out_of_service`/`archived` **أو** كان عليها طلب
    صيانة مفتوح حاجب — حتى لو أعاد أحدهم حالتها يدويًا.
  - الإكمال دون إتاحة يعيد غرفة `cleaning` إلى `dirty` (تبقى ظاهرة كغير جاهزة؛
    الإتاحة قرار صريح دائمًا). إلغاء مهمة جارية يعيد `cleaning` إلى `dirty`
    كذلك كي لا تعلق الغرفة في حالة قديمة.
- **الصيانة**:
  - عند الإنشاء (أو التعديل) بطلب حاجب تتحول الغرفة إلى `maintenance` أو
    `out_of_service` (بملاحظة تحمل رقم الطلب — يفي بشرط Phase 5 للملاحظة).
  - **لا تحرير تلقائيًا أبدًا**: `resolved` لا يغيّر الغرفة؛ `close/` يطلب
    قرارًا صريحًا `room_next_status`: `keep` / `dirty` / `available`.
    و`available` تُرفض إذا كان على الغرفة طلب حاجب آخر ما يزال مفتوحًا.
  - إلغاء طلب حاجب يعيد الغرفة إلى `dirty` (لا إلى `available` — تحتاج فحصًا)
    فقط إذا لم يبقَ طلب حاجب آخر.
  - الغرف `archived` خارج التدفق: اختيار حالة تالية لها يرفض بوضوح.

## تكامل المغادرة (Phase 7)

منطق Phase 7 لم يُمس: المغادرة ما تزال تجعل الغرفة `dirty` عبر المسار المضبوط.
**أُضيف auto-create**: بعدها مباشرة يستدعي `CheckOutService` دالة
`create_checkout_cleaning_task(stay)` التي تنشئ مهمة `checkout_cleaning`
واحدة كحد أقصى لكل إقامة (فحص idempotent بـ exists() على (الإقامة، النوع)).
تعمل داخل نفس المعاملة عمدًا: لا اعتماديات خارجية يمكن أن تفشل، وفشل نظري
فيها يعني مشكلة قاعدة بيانات تُفشل المغادرة كلها بأمان بدل ترك حالة نصفية.
مغطاة باختبارين (الإنشاء + عدم التكرار).

## الصلاحيات

ثلاثة أقسام باتت كاملة في السجل المركزي (`apps/rbac/registry.py`):

- `housekeeping`: view / create / update / cancel / status_update / assign
- `maintenance`: view / create / update / cancel / status_update / assign / close
- `lost_found`: view / create / update / status_update / close

المدير يملك الكل تلقائيًا؛ الموظف يحتاج منحًا صريحًا لكل عملية؛ صلاحيات قسم
لا تفتح قسمًا آخر؛ مالك المنصة بلا عضوية ممنوع؛ العزل بين الفنادق كما هو
(كل الاستعلامات مقيدة بـ hotel + كشوف 404 للمراجع الأجنبية).
نظرة `overview/` العامة تكفيها **أي** من صلاحيات العرض الثلاث (فئة مخصصة —
تركيب `|` القياسي كان سينفجر لأن صلاحياتنا ترفع استثناء بدل إرجاع False).

## الفندق المعلّق

القراءة (القوائم/التفاصيل/النظرة العامة) متاحة؛ وكل كتابة — إنشاء/تعديل/حالة/
إسناد/إكمال/حل/إغلاق/إلغاء/مطالبة/تسليم/إتلاف — ترفض بـ `hotel_suspended`
(نفس حارس Phase 8/9 قبل أي منطق).

## الواجهة

`/hotel/operations` بخمسة تبويبات: نظرة عامة (7 بطاقات WorkflowCard)،
التنظيف، الصيانة، المفقودات (جداول + فلاتر + مودالات إجراء)، ولوحة حالة
الغرف (6 أعمدة حسب الحالة مع آخر مهمة/طلب مفتوح وأزرار سريعة تستدعي المسارات
المضبوطة فقط). الإسناد في الواجهة «أسند إليّ» عمليًا — لا يوجد endpoint لسرد
أعضاء الفندق بعد (إدارة الموظفين مرحلة لاحقة)، والباكند يدعم الإسناد لأي عضو.
كل النصوص من قواميس ar/en/tr (تكافؤ كامل)، RTL/LTR، ومكونات مركزية فقط.

## المؤجل عمدًا

Shifts وStaff scheduling وPayroll؛ Daily Close؛ التقارير المتقدمة؛
Inventory/Stock وPurchasing وSupplier management وLaundry inventory؛
الإشعارات (WhatsApp/Email)؛ تطبيق الجوال؛ طلبات النزلاء العامة وQR tasks
وأجهزة IoT؛ صور/ملفات/باركود للمفقودات؛ اختيار أي عضو من قائمة عند الإسناد
في الواجهة (بانتظار endpoint أعضاء الفندق).
